# HtyBox 双端方案总体设计（手机配对 + iOS 远程操控终端）

> 状态：设计稿 v1（2026-06-25）。参考实现：paseo（getpaseo/paseo，本地克隆 `G:\hty_workflows\paseo`）。
> 本文是三工程的**总纲**，落在 `HtyBox_link/Document/`。各端实现细节后续在各自 `Document/` 展开。

## 0. 背景、目标与已定决策

**目标**：给 HtyBox 增加「手机配对 + iOS 应用远程实时操控终端」的能力——在电脑上跑的终端/agent，能从手机上实时查看并操控，效果对标 paseo。

**三个工程**（用户已建好目录）：

| 工程 | 角色 | 技术栈 |
|---|---|---|
| `HtyBox/`（现有） | **Host**：持有 PTY/终端，对外暴露网络协议 | Tauri 2 + Rust + React + xterm.js |
| `HtyBox_link/` | **连接层**：协议定义 + relay 中继服务 | Rust crate + TS 绑定 + 语言中立 spec |
| `HtyBox_ios/` | **iOS 客户端**：远程连接并操控终端 | Tauri Mobile（复用 HtyBox 的 React+xterm 前端） |

**两个已拍板的关键决策**：
1. **iOS 技术栈 = Tauri Mobile**：iOS app 也用 Tauri，WebView 内**复用 HtyBox 现有的 React + xterm 前端**，仅把数据源从「本地 Tauri IPC」换成「远程 WS 客户端」。技术栈统一、复用最大；代价是 Tauri iOS(v2 mobile) 较新、需踩 App Store 分发与扫码等原生能力。
2. **远程范围 = 含 E2E relay**：除局域网直连外，提供经 **自托管 relay 中继** 的远程访问（蜂窝/异地可用），relay 只转发 E2E 密文、看不到明文。

## 1. paseo 参考结论（作为设计依据）

paseo 是 **daemon + 多客户端 + 共享 protocol + relay** 架构，关键事实（已通读其 `docs/` 与 `packages/` 核实）：

- **拓扑**：daemon 监听 `127.0.0.1:6767`(WebSocket)；三条连接路径——本地直连 / LAN 直连(`0.0.0.0`) / **relay 远程**。所有客户端（桌面 Electron、移动 Expo、CLI）都是协议客户端，桌面不特殊。
- **协议**：单条 WS **混合两类帧**——文本 `JSON`（RPC/事件）+ 二进制（终端 I/O）。RPC 用 dotted-namespace + `.request`/`.response` 配对；`hello` 握手后 daemon 回 `server_info`，内含 `features.*` **能力位**用于版本协商；严格**向后兼容**（新字段必 optional、不删字段、COMPAT 标记 6 个月清理）。
- **终端二进制帧**：`[opcode(1B)][slot(1B)][payload]`，opcode = Output/Input/Resize/Snapshot/Restore；slot(0–255) 支持单连接多终端复用。
- **终端同步**：daemon 持 PTY(node-pty) + headless xterm + scrollback(1000 行)；每次输出带 `revision` 序列号；新客户端订阅时按需发 **snapshot/restore**（历史网格）再续接实时流，并按 `revision <= replayRevision` **去重**避免重复。性能上 PTY 输出经 **leading+trailing 两段合并(5ms)**，背压依据 **transport bufferedAmount** 而非输出体积；慢客户端才回退快照。
- **配对/鉴权**：daemon 持 **Curve25519 密钥对**；二维码/链接编码 `{v, serverId, daemonPublicKeyB64, relay{endpoint,useTls}}`（`#offer=base64url(json)`，fragment 不上送服务器）。客户端用 daemon 公钥做 **ECDH** 得 `shared_key`，**XSalsa20-Poly1305(NaCl box)** 加密。**公钥即信任锚**——不知公钥者无法握手；无显式"设备白名单"，配对凭证客户端保存后可无限复用。本地直连可选 Bearer 密码；**DNS rebinding** 用 Host 名白名单防护。
- **relay**：无状态转发器，按 `serverId` + `connectionId` 路由；E2E 在两端，relay 只转发密文、改不了/读不了。daemon 与手机各自 WS 连到 relay 的 `/session/{serverId}` 找到彼此。
- **客户端复用**：共享 `client` 库封装连接/重连/RPC；终端渲染在移动端是 **xterm.js 跑在 WebView** 里（桌面 web 端直接 xterm DOM）。Electron 桌面只是 app 的 web 构建外壳 + 可托管 daemon 子进程。

> 详尽机制（含文件路径、schema 摘录）见本仓 `Document/02-paseo参考详解.md`（后续补）。本节只取对 HtyBox 设计有约束力的结论。

## 2. HtyBox 现状与差距

**现状**（已核实 `src-tauri/src/pty.rs` 等）：
- `PtyManager`：`Mutex<HashMap<TermId, PtySession>>`，**同步后台读线程**把原始字节送到**单个** Tauri `Channel<Vec<u8>>`（单消费者=本地前端）；有 `spawn/write/resize/close`。
- **无** scrollback 缓冲、**无** revision 序列号、**无** 多订阅广播、**无** 背压；输出与单一前端强绑定。
- HTTP 仅 `tiny_http`（同步、`127.0.0.1:0` 随机端口，给 MCP broker 用）；**无 tokio 直接依赖**（updater 间接引入了 tokio 生态）。
- 前端：React + `@xterm/xterm`，经 `terminalEngine.ts` 把 xterm 实例与 dockview 解耦。

**要补的差距**：
1. 一个 **WS Host**（暴露协议给远程客户端；本地前端也可走同协议）。
2. 终端层 **多路化**：服务端 scrollback + revision + 广播 + snapshot/restore + 背压。
3. **配对 / 鉴权 / E2E**（Curve25519 + 会话加密）。
4. **relay** 自托管中继（远程访问）。
5. **iOS 客户端**（Tauri Mobile，复用前端）。

## 3. 总体架构

### 3.1 Host 形态：内嵌进 HtyBox 桌面进程（不另起独立 daemon）

paseo 把 daemon 拆成独立常驻进程（因为它要同时服务 web/CLI/桌面多种客户端、且 daemon 可无 UI 运行）。HtyBox **v1 不拆**：在现有 Tauri Rust 进程内**内嵌一个 WS Host**。理由：契合 HtyBox 单进程现状、改造面小、"桌面应用本身即 Host"。代价：关掉桌面=Host 下线（可接受；后续要"无界面常驻"再抽 daemon）。

**统一终端核心**：把终端能力下沉为一个 `terminal core`（持 PTY + scrollback + revision + 广播）。**本地前端与远程客户端都作为该核心的订阅者**：
- 本地前端：可继续走 Tauri IPC（in-process 订阅），也可走本地 WS（`127.0.0.1`）。**推荐本地也走 WS 协议**——让"本地前端"和"远程手机"走同一套订阅/快照/revision 路径，消除两套代码，顺带支持桌面多窗口共享同一终端。
- 远程客户端：经 WS（LAN 直连或 relay）。

### 3.2 连接三路径

```
                         ┌─────────────────────────── HtyBox 桌面进程 (Host) ───────────────────────────┐
                         │  terminal core: PtyManager + scrollback + revision + broadcast               │
                         │        ▲ in-proc 订阅                         ▲ WS 订阅                        │
   本地 React 前端 ──────┼────────┘                                       │                              │
   (Tauri WebView)       │                          WS Host (tokio+axum, 监听 127.0.0.1[+0.0.0.0])      │
                         │                                                │           │                  │
                         └────────────────────────────────────────────────┼───────────┼──────────────────┘
                                                                          │ LAN 直连   │ 反连 relay
                                                                          ▼           ▼
                                                              ┌─ 同 WiFi 手机 ─┐   ┌── relay 中继(自托管) ──┐
                                                              │  HtyBox_ios   │   │ 按 serverId+connId 转发 │
                                                              └───────────────┘   └───────────▲────────────┘
                                                                                              │ 异地手机(蜂窝)
                                                                                       HtyBox_ios (E2E)
```

- **本地直连**：`ws://127.0.0.1:<port>`，最高信任（同机隔离），可不加密。
- **LAN 直连**：Host 监听 `0.0.0.0:<port>`；二维码编码 LAN IP+端口+公钥；手机同 WiFi 直连。公钥握手 + 可选会话加密。
- **relay 远程**：Host 主动反连 relay（出站，绕过 NAT/防火墙）；手机也连 relay；relay 按 `serverId`+`connectionId` 撮合；**E2E 强制**，relay 看不到明文。

### 3.3 三工程依赖关系

```
HtyBox_link (协议 spec + Rust crate + TS 绑定)  ← 单一事实来源
   ├── HtyBox 桌面 Host        依赖 Rust crate（帧编解码/握手/E2E）
   ├── HtyBox 前端 / HtyBox_ios 依赖 TS 绑定（同一套 schema/客户端）
   └── relay 服务              依赖 Rust crate（仅需信封/路由层，不碰 E2E 内层）
```

### 3.4 关键设计决策记录（ADR 摘要）

| # | 决策 | 取舍 |
|---|---|---|
| A1 | Host 内嵌桌面进程，不拆独立 daemon（v1） | 改造小；代价=关桌面则 Host 下线 |
| A2 | 本地前端也走协议（推荐本地 WS） | 一套代码覆盖本地+远程；初期可先保留 IPC，逐步迁移 |
| A3 | WS Host 用 `tokio` + `axum`（WS+HTTP） | 标准成熟；tokio 已在依赖树；与现有 `tiny_http` broker 暂并存，后续可统一 |
| A4 | 协议 = WS 混合 JSON-RPC + 二进制终端帧 | 直接对标 paseo，已验证 |
| A5 | E2E = Curve25519 ECDH + XSalsa20-Poly1305 | 同 paseo；Rust 用 `crypto_box`/`x25519-dalek`+`xsalsa20poly1305` 或 libsodium 绑定 |
| A6 | iOS = Tauri Mobile 复用 React 前端 | 复用最大；代价=Tauri iOS 成熟度风险 |
| A7 | 向后兼容：`features` 能力位 + COMPAT 纪律 | 桌面/手机版本可不同步 |

## 4. HtyBox_link：协议层设计

### 4.1 传输与分帧
- 单条 **WebSocket** 同时承载两类帧：
  - **文本帧 = JSON**：RPC 请求/响应、服务端推送事件、握手、配对、E2E 握手（明文阶段）。
  - **二进制帧 = 终端 I/O**：`[opcode:1B][slot:1B][payload...]`。
- relay/E2E 开启后：**JSON 与二进制都被整体加密**为 `[nonce:24B][ciphertext]` 后再以 WS 二进制帧发出（内层类型由解密后首字节/字段区分）。本地直连可不加密。

### 4.2 握手与能力协商
客户端连上后先发 `hello`，Host 回 `server_info`：
```jsonc
// → client hello
{ "type":"hello", "clientId":"<uuid>", "clientType":"ios|desktop|cli",
  "protocolVersion":1, "appVersion":"1.0.0", "capabilities":{ } }
// ← host server_info
{ "type":"server_info", "serverId":"<uuid>", "version":"1.0.0",
  "features": { "terminalRestore": true, "pairing": true /*, COMPAT 标记*/ } }
```
- **向后兼容铁律**（照搬 paseo）：新增字段必 `optional`+默认；不删字段、不收窄类型、不翻 optional→required；新能力进 `features.*`，客户端检测到才启用、否则提示"请升级 Host"；每个兼容 shim 加 `// COMPAT(name): added vX, drop after <date>`。

### 4.3 JSON-RPC 命名与形态
- 命名：`域.操作.request` / `域.操作.response`（dotted namespace）。错误用 `rpc_error{requestId,error,code}`。
- 三类消息：**请求**(带 `requestId`)、**响应**(回 `requestId`)、**服务端推送事件**(无 `requestId`)。
- v1 最小 RPC 域：
  - `host.info` / `host.workspaces.list`（列工作区/cwd）
  - `terminal.list` / `terminal.create` / `terminal.subscribe` / `terminal.unsubscribe` / `terminal.kill` / `terminal.rename`
  - `pairing.offer.get`（Host 出二维码/链接）/ `pairing.devices.list` / `pairing.devices.revoke`
  - 推送：`terminal.exit`、`host.workspaces.update`、(后续)`agent.*`

### 4.4 终端二进制帧
| opcode | 值 | 方向 | payload |
|---|---|---|---|
| Output | 0x01 | Host→Client | 原始 PTY 字节（UTF-8 流） |
| Input | 0x02 | Client→Host | 按键/粘贴字节 |
| Resize | 0x03 | Client→Host | JSON `{cols,rows}` |
| Snapshot | 0x04 | Host→Client | JSON 网格快照 + `revision` |
| Restore | 0x05 | Host→Client | ANSI 序列形式的历史重放 + `revision` |

- **slot**：单连接内多终端复用（订阅时 Host 分配 0–255）。
- **revision**：每次终端输出递增的序列号，贯穿快照与增量；客户端按 `revision <= 快照revision` 去重。

### 4.5 配对 offer（二维码/链接）
```jsonc
// 编码进 QR / 链接：htybox://pair#offer=base64url(json)
{ "v":1, "serverId":"<uuid>", "hostPublicKeyB64":"<curve25519 pub>",
  "lan": { "host":"192.168.1.23", "port":6767 },        // LAN 直连候选(可空)
  "relay": { "endpoint":"relay.htybox.xxx:443", "useTls":true } } // relay 候选(可空)
```
- 客户端保存 `serverId + hostPublicKeyB64 + 连接候选`，下次免扫码重连；可存多个 Host、按延迟自适应选路（探测 LAN vs relay）。

### 4.6 E2E 加密
- 算法：**Curve25519 ECDH → shared_key → XSalsa20-Poly1305(NaCl box)**。
- 握手（明文阶段，可经 relay 透传）：
  ```
  Client → {"type":"e2ee_hello","key":"<client 临时公钥B64>"}
  Host   → {"type":"e2ee_ready"}     // 双方各自 ECDH 得相同 shared_key
  之后所有帧 = base64/二进制( [nonce24][ciphertext] )
  ```
- **安全性**：relay 不知 host 私钥 → 算不出 shared_key → 既读不了也伪造不了（NaCl 认证加密，篡改即解密失败）。
- Rust 实现候选：`crypto_box`(纯 Rust, x25519+xsalsa20poly1305) 或 `libsodium-sys`/`sodiumoxide`。客户端(TS) 用 `tweetnacl`/`libsodium-wrappers`。**两端务必跑同一套测试向量**确认 ECDH 双向一致、篡改解密失败。
- 私钥持久化：`<config>/host-keypair.json`（权限 0600 / Windows ACL），内存中用 `zeroize` 清零。

### 4.7 relay 协议（信封层，relay 只看这层）
- Host 反连：`wss://relay/session/{serverId}`（控制通道），发 `{"type":"sync"}`。
- 手机连：`wss://relay/session/{serverId}/{connectionId}`，发 `e2ee_hello`（被透传到 Host）。
- relay 按 `serverId` 找到 Host 控制通道、按 `connectionId` 建数据转发；`connected`/`disconnected` 通知 Host。
- relay **逐字节转发**密文，不解析内层；只需信封 crate，不依赖 E2E 内层。

### 4.8 HtyBox_link 工程形态
- `crates/htybox-link`（Rust）：帧编解码、hello/features、E2E、offer 编解码 —— Host 与 relay 依赖。
- `ts/`（TS 绑定）：与 Rust 同源的消息类型 + 轻量校验 + `DaemonClient` 客户端封装 —— HtyBox 前端 与 HtyBox_ios 依赖。
- `spec/`：语言中立协议文档（本设计的 §4 落为正式 spec），作为单一事实来源；Rust/TS 两侧实现都对齐它。
- 版本：`protocolVersion` 单调递增；破坏性变更走 `features` 协商，不破坏旧端解析。

## 5. HtyBox 桌面 Host 改造

### 5.1 新增 WS Host
- 新依赖：`tokio`(rt-multi-thread)、`axum`(含 ws feature) 或 `tokio-tungstenite`。与现有同步 `tiny_http` broker **暂并存**（broker 仍 `127.0.0.1:0` 给 MCP 用）；后续可把两者统一到一个 axum 服务。
- 监听：默认 `127.0.0.1:<port>`（本地）；开启"允许局域网/远程"时加 `0.0.0.0`。端口可固定（如 6767，便于二维码）或随机+写入 offer。
- 在 Tauri `setup` 起 tokio runtime + axum server，与 PtyManager 共享 `terminal core`（`Arc`）。
- 安全：Host 名白名单（防 DNS rebinding）、Origin 校验；远程一律要求 E2E；本地可选 Bearer 密码。

### 5.2 终端层重构（terminal core）
把 `pty.rs` 升级为可多路广播 + 历史重放（对标 paseo `terminal.ts`）：
- **scrollback 环形缓冲**：`VecDeque<(revision, Vec<u8>)>`，容量按行数/字节上限（如 ~1MB 或 N 块）。
- **revision**：`AtomicU64`，每次读到 PTY 输出 +1，随广播与快照携带。
- **广播**：`tokio::sync::broadcast`（或自维护 `Vec<Subscriber>`）；本地前端订阅 + 每个 WS 客户端订阅各为一个 receiver。
- **snapshot/restore**：维护一个 headless 终端状态（可选先做简单版：直接重放 scrollback 原始字节作为 Restore；进阶版接 `vte`/headless 解析出网格快照）。
- **coalescing + 背压**：输出按 ~5ms leading+trailing 合并；按 WS `bufferedAmount`(或发送队列长度) 判断慢客户端，超阈值回退快照对齐。
- 接口（示意）：`create(opts) -> termId`、`subscribe(termId, restoreMode, fromRevision) -> (snapshot?, stream)`、`input(termId, bytes)`、`resize(termId, cols, rows)`、`kill(termId)`。

### 5.3 本地前端改造（渐进）
- 阶段 a（先不动）：本地前端继续走 Tauri IPC，但底层改调 terminal core（带 revision/scrollback），保证两条路径数据一致。
- 阶段 b（推荐目标）：本地前端改成连 `ws://127.0.0.1` 的协议客户端（复用 HtyBox_link TS `DaemonClient`）。好处：本地与远程**完全同构**、桌面多窗口天然共享终端、只维护一套订阅/快照逻辑。
- `terminalEngine.ts` 改为消费"带 revision 的输出流 + 首次 snapshot/restore"，并做客户端去重。

### 5.4 配对 UI（对标截图的「连接 / 配对设备」）
在 HtyBox 设置内新增「连接」页（自定义弹窗/页，遵守奶油主题、禁原生对话框）：
- 显示当前监听（如 `TCP 127.0.0.1:6767`、LAN `192.168.x.x:6767`、relay 状态）。
- **配对设备**：展示二维码 + 可复制链接（`htybox://pair#offer=...`）；列出已配对设备、支持「移除」。
- 开关：是否允许 LAN、是否启用 relay、（可选）本地密码。

### 5.5 密钥 / serverId / relay 连接管理
- 首次启动生成 Curve25519 密钥对 + `serverId(uuid)`，持久化到配置目录（0600）。
- relay：Host 出站反连 relay 控制通道；断线指数退避重连；UI 显示 relay 在线/离线。

### 5.6 复用现有资产
- 现有 `broker.rs`(tiny_http) 的"宿主免 token 读快照"模式可借鉴到 Host 的 snapshot 思路。
- 现有崩溃自愈(`terminal-exit` 事件)、agent 身份(env 注入) 等保持；远程客户端看到的也是同一批终端。

## 6. relay 中继服务（自托管）

- **形态**：独立小型 Rust 服务（`axum` + `tokio-tungstenite`），无状态、只转发；可放进 `HtyBox_link/relay/` 或独立部署。
- **路由**：内存 `Map<serverId, 控制连接>` + `Map<(serverId,connectionId), 数据转发>`；按 URL 路径解析 `serverId`/`connectionId`。
- **零明文**：只转发密文字节，不解析内层；唯一信任假设=不伪造路由。
- **部署**：一台有公网的小 VPS（或 Cloudflare Workers + Durable Objects，对标 paseo）。需 TLS(wss)；域名写进 offer。
- **健壮性**：心跳 ping/pong、连接配额、`serverId` 抢占保护（同 serverId 新控制连接顶替旧的需校验）。
- **隐私**：relay 不需要也不应保存任何会话内容；日志只记连接元数据。

## 7. HtyBox_ios：Tauri Mobile 客户端

### 7.1 形态与复用策略
- HtyBox_ios 是 **Tauri 2 iOS app**：WebView 内跑 React 前端，**复用 HtyBox 现有终端渲染 + 部分 UI**。
- 与桌面的差别：**数据源不同**——桌面前端订阅本地 terminal core；iOS 前端用 HtyBox_link 的 TS `DaemonClient` 连远程 Host（LAN 或 relay）。
- **抽出共享前端包**（关键工程动作）：把"终端视图组件 + 协议客户端 + 终端状态/去重"从 HtyBox 抽成一个可被桌面与 iOS 共同依赖的包（如 `packages/terminal-ui` 或 monorepo 化）。桌面注入"本地 transport"，iOS 注入"远程 WS transport"。
- iOS 端 Rust 侧（Tauri mobile）通常很薄：主要做原生桥（扫码、安全存储、可能的 WS/网络若放 Rust 层）；也可让 WS 直接在 WebView 的 JS 里跑（`WebSocket` API），Rust 只提供扫码与安全存储插件。

### 7.2 iOS 特有能力
- **扫码配对**：相机扫二维码 → 解析 offer → 保存 Host。Tauri 有 `tauri-plugin-barcode-scanner`（移动端），优先用它；否则自写原生插件。
- **配对凭证安全存储**：iOS Keychain（经 Tauri secure-store/自写插件）保存 `serverId + hostPublicKey + 连接候选`。
- **网络/后台**：WS 重连、息屏/切后台再回前台的重连与历史重放（依赖 §4 revision/snapshot）。
- **触屏终端体验**：软键盘 + 特殊键条（Esc/Ctrl/Tab/方向键/管道符等），手势滚动 scrollback；xterm 在移动 WebView 的输入处理参考 paseo `terminal-pane`。

### 7.3 最小屏幕集（v1）
1. **配对/欢迎**：扫码 / 粘贴链接 / （LAN 直填 IP:端口）。
2. **Host 列表**：已保存 Host，连接状态，切换/删除。
3. **Host 主页**：列出该 Host 的终端（和工作区）。
4. **终端视图**：xterm + 软键盘 + 特殊键条 + 工具栏（新建/重命名/关闭、断线重连提示）。

### 7.4 构建 / 发布
- `tauri ios init` 生成 Xcode 工程；本地 `tauri ios dev`（模拟器/真机）；`tauri ios build` 出包。
- 需 Apple 开发者账号、签名、TestFlight 分发。`ITSAppUsesNonExemptEncryption` 因含加密需如实声明（自用 E2E，多走 exempt 声明）。
- 风险见 §10：Tauri iOS 插件生态与稳定性需早验证（先做"扫码+连一个 Host+看一个终端"的最小贯通）。

## 8. 端到端时序（关键路径）

**(1) 配对**
```
桌面 Host: 设置→连接→「配对设备」→ pairing.offer.get → 渲染 QR(htybox://pair#offer=...)
iOS:      扫码 → 解析 offer → 存 {serverId, hostPublicKey, lan?, relay?} → 列入 Host 列表
```
**(2) 连接 + E2E（以 relay 为例）**
```
Host(常驻): 反连 wss://relay/session/{serverId} 控制通道, 等待
iOS: 连 wss://relay/session/{serverId}/{connId}
iOS → e2ee_hello{client临时公钥}  (relay 透传) → Host
Host: ECDH→shared_key, 回 e2ee_ready (透传) → iOS: ECDH→同一 shared_key
之后: hello/server_info(features) 在加密信道内完成
```
**(3) 订阅终端 + 历史重放**
```
iOS → terminal.subscribe{termId, restore:"visible-snapshot", scrollbackLines:200}
Host → Snapshot/Restore 帧(含 revision R0) → iOS 渲染历史
Host → Output 帧(revision>R0) 实时流 → iOS 去重(<=R0 跳过)后 xterm.write
```
**(4) 输入 / resize**
```
iOS 按键 → Input 帧(slot) → Host PtyManager.write → PTY 回显 → Output 广播给所有订阅者(本地+iOS)
iOS viewport 变化 → Resize 帧 → Host PtyManager.resize（last-interacting-client-wins，不广播 resize）
```

## 9. 里程碑与落地顺序

每步**独立可验收**，先把"本地自连"打通再上手机，最后接 relay：

| 里程碑 | 内容 | 验收 |
|---|---|---|
| **L1 协议骨架** | `HtyBox_link`：Rust crate + TS 绑定的帧编解码/hello/features/offer/E2E；spec 文档 | 单测：编解码往返、ECDH 双向一致、篡改解密失败 |
| **L2 Host WS + 终端多路** | terminal core(scrollback+revision+广播+snapshot/restore+背压)；axum WS Host；**本地前端改走 `127.0.0.1` WS** | 桌面用 WS 路径正常开/输入/输出终端；开两个本地窗口共享同一终端、各自历史重放正确 |
| **L3 配对 + 鉴权** | 密钥/serverId 持久化；配对 UI(二维码/链接/已配对管理)；LAN 直连 + E2E | 同 WiFi 另一台机器(或浏览器原型)扫码连上、E2E 生效、未配对被拒 |
| **L4 relay** | relay 服务(部署一台)；Host 反连；手机经 relay 连 | 关掉同网，手机走蜂窝经 relay 操控终端；relay 抓包只见密文 |
| **L5 iOS 客户端** | HtyBox_ios(Tauri Mobile)：抽共享前端包；扫码/Host 列表/终端视图；Keychain 存凭证 | 真机扫码→连 Host→实时看并操控终端、历史重放、断线重连 |
| **L6 打磨** | 背压/重连/多端一致/软键盘特殊键/性能；向后兼容回归 | 弱网/长输出/多客户端压测达标 |

> 建议 L2 之前先把 L1 的 spec 与 crate 定稿——协议是三端契约，定得稳后并行开工。

## 10. 风险与取舍

| 风险 | 说明 | 缓解 |
|---|---|---|
| **Tauri iOS 成熟度** | v2 mobile 较新，扫码/Keychain/WS 后台等插件生态待验证 | L5 先做"扫码+连一个 Host+一个终端"最小贯通验证可行性，再铺开；备选退回 Expo/RN |
| **relay 运维** | 需公网 VPS + TLS + 长连可靠性 | 先用便宜 VPS/Cloudflare；relay 无状态、易重启；LAN 路径不依赖 relay 仍可用 |
| **E2E 正确性** | 自己接密码学易错 | 用成熟库 + 固定测试向量 + 双向一致性测试；不自造算法 |
| **终端多路/ConPTY** | Windows ConPTY 多客户端、快照、历史重放复杂；叠加既有终端叠影技术债 | 快照初版用"重放 scrollback 原始字节"避免网格解析；先解决既有渲染技术债(见 .claude/bugs) |
| **本地前端迁到 WS** | 改动面大、可能引入回归 | 渐进：先 terminal core 内核统一(IPC 仍可用)，再切 WS；保留回滚点 |
| **向后兼容** | 桌面/手机版本不同步 | 严格 features 协商 + COMPAT 纪律(照搬 paseo) |
| **性能/弱网** | 长输出、蜂窝高延迟 | coalescing + 背压 + 快照对齐；revision 去重防重放 |

## 11. 未决 / 待用户拍板

- **relay 部署位置**：自建 VPS vs Cloudflare Workers(对标 paseo) vs 暂不部署(先只做 LAN，relay 留 L4)。
- **是否同时出 Android**：Tauri Mobile 也能出 Android，几乎零额外协议成本；是否纳入。
- **本地前端是否切 WS**：是否执行 A2 的阶段 b（推荐但可延后）。
- **端口策略**：固定 6767 vs 随机端口写入 offer。
- **本地密码**：本地直连是否额外要 Bearer 密码（深度防御）。
